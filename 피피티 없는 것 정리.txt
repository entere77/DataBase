트랜잭션(transaction)(거래)의 개념
"하나의 작업을 수행하는데 필요한 데이터베이스 연산들을 모아놓은 것"
논리적인 작업의 단위
작업 수행에 필요한 SQL문들의 모임
장애 발생 시 복구 작업이나 병행 제어 작업을 위한 중요한 단위로 사용됨
데이터베이스의 무결성과 일관성을 보장하기 위해 작업 수행에 필요한 연산들을 하나의 트랜잭션으로 제대로 정의하고 관리해야 함

트랜잭션의 연산
commit 연산 : 트랜잭션이 성공적으로 수행되었음을 선언(작업 완료)
rollback 연산 : 트랜잭션을 수행하는 데 실패했음을 선언(작업 취소)


"""트랜잭션의 상태"""
트랜잭션의 다섯 가지 상태

"상태 다이어 그램"
트랜잭션 수행 시작 -> 활동 상태 -> 부분 완료 (상태) -> 실패(상태)   (rollback 연산) -> 철회 (상태)
			      -> 실패 상태          (commit 연산)-> 완료 (상태)



회복(recovery)
장애가 발생했을 때 데이터베이스를 장애가 발생하기 전의 일관된 "상태"로 복구시키는 것
트랜잭션의 특성을 보장하고, 데이터베이스를 일관된 상태로 유지하기 위해 필수적인 기능
회복 관리자가 담당
장애 발생을 탐지하고, 장애가 발생하면 데이터베이스를 기록한다.

""로그"" 회복 기법 - 즉시 갱신 회복 기법
트랜잭션 수행 중에 데이터 변경 연산의 결과를 데이터베이스에 즉시 반영
장애 발생에 대비하기 위해 데이터 변경에 대한 내용을 로그 파일에 기록
데이터 변경 연산이 실행되면, 로그 파일에 로그 레코드를 먼저 기록한 후 데이터 베이스에 변경 연산을 반영
장애 발생 시점에 따라 redo나 undo 연산을 실행해 데이터베이스를 복구

"트랜잭션 스케줄"
트랜잭션에 포함되어 있는 연산들을 수행하는 순서

트랜잭션 스케줄의 유형

1.직렬 스케줄 : 인터리빙 방식을 이요하지 않고 트랜잭션별로 연산들을 순차적으로 실행시키는 것
2. 비직렬 스케줄 : 인터리빙 방식을 이용하여 트랜잭션들을 병행해서 수행시키는 것
3. 직렬 가능 스케줄 : 직렬 스케줄과 같이 정확한 결과를 생성하는 비직렬 스케줄


로킹 기법
한 트랜잭션이 먼저 접근한 데이터에 대한 연산을 끝낼 때까지는 다른 트랜잭션이 그 데이터에 접근하지 못하도록 상호배제함

방법
병행 수행되는 트랜잭션들이 같은 데이터에 동시에 접근하지 못하도록 lock과 unlock을 사용한다.


교착상태
locking이 되어 있는 상태
트랜잭션들이 상대가 독점하고 있는 데이터에 unlock 연산이 실행되기를 서로 기다리면서 수행을 중단하고 있는 상태

교착 상태가 발생하지 않도록 예방하고나, 발생 시 빨리 탐지하여 필요한 조치를 취해야 함


덤프 : 데이터베이스 전체를 다른 저장 장치에 주기적으로 복사하는 방법
로그 : 이전값과 변경한 이후 값을 파일에 저장하는 것

데이터베이스 시스템
데이터베이스 관리 시스템을 서술하시오

데이터베이스는 조직에 필요한 정보를 얻기 위해 논리적으로 연관된 데이터를 모아 테이블로 통합해 놓은 것을 말한다.

데이터베이스 시스템은 데이터의 검색과 변경을 주로 수행한다. 즉, SQL을 수행한다.

데이터베이스 관리 시스템은 데이터베이스 관리와 사용자의 데이터 처리 요구를 수행한다.


정규화를 하는 이유
정규화 3가지 나온다.
1차는 중복성 제거
2차부터 쪼갠다. 종속성을 기준으로. 판단해서.


정규화란 테이블을 분해하는 과정이다.

삽입 이상 : 새 데이터를 삽입하기 위해서 불필요한 데이터도 함께 삽입해야 하는 문제
갱신 이상 : 중복 투플 중 일부만 변경하여 데이터가 불일치 되는 모순의 문제
삭제 이상 : 투플을 삭제하면 꼭 필요한 데이터까지 함께 삭제되는 데이터 손실의 문제

정규화가 필요한 이유 : 이상현상이 발생하면 공간 낭비 및 부작용이 발생하기 때문이다.
